Concepts Primer — From Zero

Before any code, here is plain-English explanation of every technology Brio uses.

What is a Database?

A database is an organised digital filing cabinet. Data is stored in tables (like spreadsheet sheets). Every row is one record; every column is one attribute. The rules about what goes in each column are called the schema.

Analogy
A school register is a database. Each page = a table. Each row = a student. Each column = a field (name, roll number, marks). The rule that no two students share a roll number is enforced by the database — not by a teacher remembering to check.

What is SQL?

SQL (Structured Query Language) is the language used to talk to a relational database. Four categories:

CategoryFull NameWhat it doesKeywords
DDLData Definition LanguageCreates / changes the structureCREATE, ALTER, DROP
DMLData Manipulation LanguageAdds / changes / deletes dataINSERT, UPDATE, DELETE
DQLData Query LanguageReads dataSELECT
DCLData Control LanguageControls permissionsGRANT, REVOKE

What is Node.js?

Normally JavaScript only runs inside a browser. Node.js is a runtime that lets you run JavaScript on a server — a computer that handles requests from browsers. Brio's entire backend is a single JavaScript file running on Node.js.

What is Express?

Express is a minimalist framework for Node.js that makes building web servers easy. It is a router: "when someone sends a POST request to /api/auth/login, run this function."

What is an API and JSON?

An API (Application Programming Interface) is a defined set of routes a server exposes for clients to call. Brio has 11 routes. Each one returns data as JSON — a text format that represents objects with key–value pairs, for example: { "token": "eyJ...", "role": "learner" }.

Analogy — Restaurant
The menu = the API (what you can order). You = the browser sending a request. The waiter = HTTP. The kitchen = the backend + database. The plate = the JSON response.

What is JWT?

JWT (JSON Web Token) is a digitally signed ID card. On login, the server creates a token containing your user ID and role, signed with a secret key. Every subsequent request includes this token in its header. The server verifies the signature to trust the contents — no database lookup needed to confirm identity on every request.

What is bcrypt?

Passwords must never be stored as plain text. bcrypt is a one-way hashing algorithm: it converts a password into a fixed-length scrambled string. The original password cannot be recovered from the hash. At login, bcrypt re-scrambles what the user typed and compares it to the stored hash.

What is Zod?

Zod is a JavaScript validation library. It defines the expected shape of incoming data and rejects anything that doesn't match — before any database query runs. Example: "email must be a valid email address, password must be at least 1 character."

Database Schema — All 13 Tables

The schema is the complete blueprint of the database. Brio's schema is in Third Normal Form (3NF) — no redundant data, no update anomalies.

What is Normalisation?

Normal FormRuleHow Brio satisfies it
1NFEvery cell holds exactly one value. No lists inside a column.Lesson tags live in a separate lesson_tag table, not as a comma-separated string in lesson.
2NFEvery non-key column depends on the whole primary key (no partial dependencies).enrollment(user_id, course_id) has composite PK; grade and progress depend on both together.
3NFNo non-key column depends on another non-key column (no transitive dependencies).tech_stack lives in course, not repeated in lesson. Course data is accessed via FK.

DDL — Creating the Database

File: sql/01_DDL.sql. Must run first. Tables are created in dependency order — you cannot create a table that references another table that doesn't exist yet.

users table — annotated line by line

CREATE TABLE users (
    user_id      INT PRIMARY KEY AUTO_INCREMENT,  -- MySQL assigns next integer; never insert manually
    f_name       VARCHAR(50)  NOT NULL,             -- first name, required
    l_name       VARCHAR(50)  NOT NULL,
    email        VARCHAR(100) NOT NULL UNIQUE,      -- DB rejects duplicates, even if app forgets
    password_hash VARCHAR(255) NOT NULL,            -- bcrypt hash — NEVER store plaintext passwords
    bio          TEXT         DEFAULT NULL,
    avatar_url   VARCHAR(255) DEFAULT NULL,
    github_url   VARCHAR(255) DEFAULT NULL,
    role         ENUM('learner','mentor','admin') NOT NULL DEFAULT 'learner',  -- only 3 valid values
    joined_at    TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,              -- auto-set on insert
    sub_tier     ENUM('starter','pro','team') NOT NULL DEFAULT 'starter'
);
Key DDL concepts in this one table
  • AUTO_INCREMENT — MySQL keeps an internal counter and increments it on every INSERT. You never supply user_id manually.
  • UNIQUE — a database-level constraint. Even a direct SQL query bypassing the application cannot insert a duplicate email.
  • ENUM — restricts the column to a fixed set. Inserting 'superuser' into role raises an error.
  • DEFAULT — if the column is omitted during INSERT, MySQL uses this value automatically.
  • NOT NULL — inserting a row without supplying this column fails immediately.

kanban_task — composite primary key explained

CREATE TABLE kanban_task (
    project_id  INT          NOT NULL,
    task_name   VARCHAR(150) NOT NULL,
    lesson_id   INT          DEFAULT NULL,   -- NULL = no skill gate required
    status      ENUM('locked','in_progress','done') NOT NULL DEFAULT 'locked',
    skill_req   VARCHAR(200) DEFAULT NULL,
    priority    ENUM('low','medium','high') NOT NULL DEFAULT 'medium',
    created_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (project_id, task_name),         -- composite: each project has its own task names
    FOREIGN KEY (project_id) REFERENCES project(project_id) ON DELETE CASCADE,
    FOREIGN KEY (lesson_id)  REFERENCES lesson(lesson_id)   ON DELETE SET NULL
);
Why a composite primary key?
Two different projects can both have a task named "Write tests". The task name alone is not globally unique. But the combination of project_id + task_name is always unique — each project owns its own set of uniquely named tasks. This is the hallmark of a weak entity: identity depends on the owner.

Foreign key actions — CASCADE vs SET NULL vs RESTRICT

ActionWhat happens when the parent row is deletedUsed in Brio where
ON DELETE CASCADEChild row is automatically deleted tooDeleting a project deletes all its kanban_tasks
ON DELETE SET NULLThe FK column in the child is set to NULLDeleting a lesson sets kanban_task.lesson_id to NULL — the task survives, just loses its gate
ON DELETE RESTRICTThe parent delete is blocked if any child existsPrevents accidental cascading data loss

submission — CHECK constraint

CREATE TABLE submission (
    submission_id INT PRIMARY KEY AUTO_INCREMENT,
    user_id     INT       NOT NULL,
    sub_date    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    lesson_id   INT       DEFAULT NULL,
    ai_score    INT       CHECK (ai_score BETWEEN 0 AND 100),  -- DB rejects 101, -5, etc.
    feedback    TEXT      DEFAULT NULL,
    code_url    VARCHAR(255) DEFAULT NULL,
    UNIQUE (user_id, sub_date),                -- no two submissions from same user in the same second
    FOREIGN KEY (user_id)   REFERENCES users(user_id)    ON DELETE CASCADE,
    FOREIGN KEY (lesson_id) REFERENCES lesson(lesson_id) ON DELETE SET NULL
);

DML — Seeding Data

File: sql/02_DML.sql. Runs after DDL. Inserts sample data in dependency order: strong entities first, junction tables last.

Inserting users with pre-hashed passwords

INSERT INTO users (f_name, l_name, email, password_hash, role, sub_tier) VALUES
('James', 'Wilson', 'james.wilson@devpath.io',
 '$2b$10$abc...',   -- bcrypt hash of 'password123'
 'learner', 'pro'),
('Admin', 'Brio', 'admin@brio.dev',
 '$2b$10$xyz...',   -- bcrypt hash of 'admin2026'
 'admin', 'team');
Why passwords are hashed in the seed file
If someone reads the SQL file, they must not see plaintext passwords. The hashes are generated offline using bcrypt (10 salt rounds) and pasted into the DML. In the running application, the registration endpoint hashes before inserting — the DB never sees plaintext.

Inserting a junction row (M:N relationship)

-- Enroll user 1 (James) into course 1 (Full-Stack Web Dev)
INSERT INTO enrollment (user_id, course_id, progress) VALUES
(1, 1, 0);
-- (user_id=1, course_id=1) is the composite PK — uniquely identifies this enrollment
-- progress starts at 0% and is updated as lessons are completed

Upsert — INSERT … ON DUPLICATE KEY UPDATE

The submit endpoint uses an upsert so re-submitting the same lesson at the same second does not hard-error:

INSERT INTO submission (user_id, sub_date, lesson_id, ai_score, feedback)
VALUES (?, NOW(), ?, ?, ?)
ON DUPLICATE KEY UPDATE
    ai_score = VALUES(ai_score),
    feedback = VALUES(feedback);

If the UNIQUE(user_id, sub_date) constraint fires, MySQL updates the existing row instead of erroring. This pattern is called an upsert (update if exists, insert if not).

The 15 Showcase Queries

File: sql/03_QUERIES.sql. These run live in the Showcase tab from the real database. Each demonstrates a specific SQL feature.

Q1 — GROUP BY + COUNT (lessons per course) 
SELECT c.title AS course_title, c.difficulty,
       COUNT(l.lesson_id) AS lesson_count
FROM course c
LEFT JOIN lesson l ON c.course_id = l.course_id
GROUP BY c.course_id, c.title, c.difficulty
ORDER BY lesson_count DESC;
Explanation
LEFT JOIN keeps courses with zero lessons. GROUP BY collapses all rows for the same course into one. COUNT counts lessons in that group. ORDER BY DESC puts the most lesson-rich courses first.
Q2 — Multi-table INNER JOIN (active projects + blueprints) 
SELECT p.proj_name, p.status, b.description AS blueprint_desc, b.difficulty_est
FROM project p
JOIN blueprint b ON p.blueprint_id = b.blueprint_id
WHERE p.status = 'active';
Explanation
INNER JOIN (just JOIN) returns only rows where a match exists on both sides. This shows active projects alongside their AI-generated blueprint descriptions — the 1:1 relationship between project and blueprint in one query.
Q4 — CASE WHEN (pivot: task status counts per project) 
SELECT p.proj_name,
       COUNT(kt.task_name) AS total_tasks,
       SUM(CASE WHEN kt.status = 'done'        THEN 1 ELSE 0 END) AS done_tasks,
       SUM(CASE WHEN kt.status = 'in_progress' THEN 1 ELSE 0 END) AS in_progress,
       SUM(CASE WHEN kt.status = 'locked'      THEN 1 ELSE 0 END) AS locked_tasks
FROM project p
LEFT JOIN kanban_task kt ON p.project_id = kt.project_id
GROUP BY p.project_id, p.proj_name;
Explanation
CASE WHEN is SQL's if/else — here it turns each status value into 1 or 0. SUM() then counts them. This "pivot" pattern converts row values into separate column counts within a single grouped query — a very common reporting technique.
Q5 — AVG + ROUND + MAX (performance leaderboard) 
SELECT CONCAT(u.f_name, ' ', u.l_name) AS full_name,
       COUNT(s.user_id)                 AS total_submissions,
       ROUND(AVG(s.ai_score), 1)         AS avg_ai_score,
       MAX(s.ai_score)                  AS best_score
FROM users u
JOIN submission s ON u.user_id = s.user_id
GROUP BY u.user_id, u.f_name, u.l_name
ORDER BY avg_ai_score DESC;
Explanation
CONCAT combines first and last name into a single string. AVG computes the mean score per user. ROUND limits to one decimal place. ORDER BY DESC puts the best performers first — this is a leaderboard.
Q6 — 3-table JOIN (full enrollment report) 
SELECT CONCAT(u.f_name, ' ', u.l_name) AS student_name,
       u.sub_tier, c.title AS course, c.difficulty,
       e.progress, e.grade
FROM enrollment e
JOIN users  u ON e.user_id  = u.user_id
JOIN course c ON e.course_id = c.course_id
ORDER BY e.progress DESC, student_name;
Explanation
The junction table enrollment sits in FROM. Both sides of the M:N relationship (users and course) are JOINed onto it. This resolves many-to-many into readable flat rows — the standard pattern for querying M:N relationships.
Q7 — 4-table JOIN with LEFT JOIN and multi-condition ON 
SELECT t.team_name,
       CONCAT(u.f_name, ' ', u.l_name) AS member_name,
       tr.role AS team_role,
       p.proj_name AS working_on
FROM team t
JOIN      team_member tm ON t.team_id  = tm.team_id
JOIN      users u        ON tm.user_id = u.user_id
LEFT JOIN team_role tr   ON (t.team_id = tr.team_id AND u.user_id = tr.user_id)
LEFT JOIN project p      ON (p.team_id = t.team_id AND p.status = 'active')
ORDER BY t.team_name, tr.role;
Explanation
LEFT JOINs on team_role and project ensure members without a role or no active project still appear (with NULLs in those columns). The multi-condition ON for project adds an extra filter — only active projects inline, without a WHERE that would exclude non-project members.
Q10 — LEFT JOIN + IS NULL (users with no submissions) 
SELECT CONCAT(u.f_name, ' ', u.l_name) AS full_name,
       u.email, u.role, u.sub_tier
FROM users u
LEFT JOIN submission s ON u.user_id = s.user_id
WHERE s.user_id IS NULL
ORDER BY u.f_name;
The NULL trick
LEFT JOIN keeps all users even with no matching submission. When no match exists, every column from the submission side is NULL. Filtering WHERE s.user_id IS NULL isolates users who have never submitted — the standard "anti-join" pattern.
Q11 — GROUP_CONCAT (tag cloud per lesson) 
SELECT c.title AS course_title, l.title AS lesson_title,
       GROUP_CONCAT(lt.tag ORDER BY lt.tag SEPARATOR ', ') AS tags
FROM course c
JOIN lesson l      ON c.course_id = l.course_id
JOIN lesson_tag lt ON l.lesson_id = lt.lesson_id
GROUP BY c.course_id, c.title, l.lesson_id, l.title
ORDER BY c.title, l.lesson_no;
Explanation
GROUP_CONCAT is MySQL-specific — it collapses multiple rows into one comma-separated string within a group. This "re-collects" the multivalued tags that were normalised into their own table, presenting them as a readable list without denormalising the schema.
Q12 — Subquery with IN (users enrolled in auth-tagged courses) 
SELECT DISTINCT CONCAT(u.f_name, ' ', u.l_name) AS full_name, u.email
FROM users u
JOIN enrollment e ON u.user_id = e.user_id
WHERE e.course_id IN (
    SELECT DISTINCT l.course_id
    FROM lesson l
    JOIN lesson_tag lt ON l.lesson_id = lt.lesson_id
    WHERE lt.tag = 'auth'
);
Explanation
The inner subquery runs first and produces a list of course_ids that contain at least one lesson tagged 'auth'. The outer query then uses IN to filter enrollments to only those courses. Subqueries compute a temporary answer that the outer query uses — think of them as nested questions.
Q14 — Nested subquery with HAVING MAX (top scorer) 
SELECT CONCAT(u.f_name, ' ', u.l_name) AS full_name,
       ROUND(AVG(s.ai_score), 1) AS avg_score
FROM users u
JOIN submission s ON u.user_id = s.user_id
GROUP BY u.user_id, u.f_name, u.l_name
HAVING AVG(s.ai_score) = (
    SELECT MAX(avg_sub)
    FROM (SELECT AVG(ai_score) AS avg_sub
          FROM submission GROUP BY user_id) AS scores
);
Explanation — three levels
Innermost: compute every user's average score. Middle: find the maximum of those averages. HAVING: keep only users whose average equals that maximum. WHERE filters raw rows; HAVING filters groups after aggregation — that distinction is critical.
Q15 — NOT IN subquery (locked tasks with unmet prerequisites) 
SELECT p.proj_name, kt.task_name, l.title AS required_lesson
FROM kanban_task kt
JOIN project p ON kt.project_id = p.project_id
JOIN lesson l  ON kt.lesson_id  = l.lesson_id
WHERE kt.status = 'locked'
  AND l.lesson_id NOT IN (
      SELECT DISTINCT lesson_id
      FROM submission
      WHERE user_id  = p.user_id
        AND ai_score >= 70
  );
Explanation
The subquery produces a whitelist of lessons the project owner has passed (score ≥ 70). NOT IN then finds locked tasks whose gating lesson is not on that whitelist — meaning the prerequisite is not yet met.

Views — The Database's Saved Reports

A view is a named, stored SELECT query that behaves exactly like a table. No data is stored in the view itself — it recomputes every time it is queried. Views encapsulate complex logic so that API endpoints can run a simple SELECT * FROM view_name WHERE … instead of a multi-table JOIN every time.

View 1 — learner_dashboard_view

Queried by: GET /api/profile/me

CREATE OR REPLACE VIEW learner_dashboard_view AS
SELECT
    u.user_id,
    CONCAT(u.f_name, ' ', u.l_name)   AS full_name,
    u.sub_tier,
    COUNT(DISTINCT e.course_id)        AS courses_enrolled,
    ROUND(AVG(e.progress), 1)          AS avg_course_progress,
    COUNT(DISTINCT s.sub_date)         AS total_submissions,
    ROUND(AVG(s.ai_score), 1)          AS avg_ai_score,
    COUNT(DISTINCT p.project_id)       AS projects_owned
FROM users u
LEFT JOIN enrollment e ON u.user_id = e.user_id
LEFT JOIN submission s ON u.user_id = s.user_id
LEFT JOIN project p    ON u.user_id = p.user_id
WHERE u.role = 'learner'
GROUP BY u.user_id, u.f_name, u.l_name, u.sub_tier;
Why COUNT DISTINCT?
Without DISTINCT, the LEFT JOIN cross-product would inflate counts. If a user has 3 enrollments and 5 submissions, the naive join produces 15 rows — COUNT would return 15, not 3. COUNT DISTINCT counts unique values only, giving the correct figures.

View 2 — project_kanban_view

Queried by: GET /api/projects

CREATE OR REPLACE VIEW project_kanban_view AS
SELECT
    p.project_id, p.proj_name, p.status AS project_status,
    CONCAT(u.f_name, ' ', u.l_name) AS owner,
    t.team_name,
    COUNT(kt.task_name)                                          AS total_tasks,
    SUM(CASE WHEN kt.status = 'done'        THEN 1 ELSE 0 END) AS done_count,
    SUM(CASE WHEN kt.status = 'in_progress' THEN 1 ELSE 0 END) AS in_progress,
    SUM(CASE WHEN kt.status = 'locked'      THEN 1 ELSE 0 END) AS locked_count,
    ROUND(
        SUM(CASE WHEN kt.status = 'done' THEN 1 ELSE 0 END) * 100.0
        / COUNT(kt.task_name), 2
    ) AS completion_pct
FROM project p
JOIN      users u          ON p.user_id   = u.user_id
LEFT JOIN team t           ON p.team_id   = t.team_id
LEFT JOIN kanban_task kt   ON p.project_id = kt.project_id
GROUP BY p.project_id, p.proj_name, p.status, u.f_name, u.l_name, t.team_name;

View 3 — skill_gate_compliance_view

Queried by: Showcase tab → View 3 button

CREATE OR REPLACE VIEW skill_gate_compliance_view AS
SELECT
    p.proj_name, kt.task_name, kt.status,
    l.title                          AS gating_lesson,
    CONCAT(u.f_name, ' ', u.l_name)  AS owner,
    CASE
        WHEN s.ai_score IS NOT NULL AND s.ai_score >= 70
             THEN 'Eligible to unlock'
        WHEN s.ai_score IS NOT NULL AND s.ai_score < 70
             THEN 'Score too low (need >= 70)'
        ELSE 'Lesson not submitted yet'
    END                              AS gate_status,
    s.ai_score                       AS lesson_score
FROM kanban_task kt
JOIN      project p   ON kt.project_id = p.project_id
JOIN      users u     ON p.user_id     = u.user_id
LEFT JOIN lesson l    ON kt.lesson_id  = l.lesson_id
LEFT JOIN submission s
          ON (s.user_id = u.user_id AND s.lesson_id = kt.lesson_id)
WHERE kt.status = 'locked';
Multi-condition LEFT JOIN on submission
The submission LEFT JOIN uses two conditions: matching the project owner's user_id AND the specific gating lesson_id. Without both, we'd show any user's score for that lesson — not the project owner's.

Triggers — Automation Inside the Database

A trigger is a stored procedure that MySQL runs automatically when a specific event (INSERT, UPDATE, or DELETE) occurs on a table. No application code calls it — the database fires it silently.

Why triggers instead of application code?
Triggers fire regardless of what causes the event — an API call, a direct MySQL Workbench query, a migration script, anything. Application code can be bypassed; database triggers cannot. Business rules enforced in triggers are guaranteed.

Trigger 1 — Auto-unlock kanban tasks on passing submission

INSERT INTO submission (ai_score=85, lesson_id=4) │ ─────────────────▼───────────────── trg_unlock_task_after_submission (AFTER INSERT ON submission) ───────────────────────────────── IF NEW.ai_score >= 70 AND lesson_id given: UPDATE kanban_task SET status = 'in_progress' WHERE lesson_id = 4 ← the lesson just passed AND status = 'locked' AND project.user_id = the submitter ───────────────────────────────── │ (status changed to 'in_progress') │ ─────────────────▼───────────────── trg_auto_ship_project fires next (AFTER UPDATE ON kanban_task) ───────────────────────────────── IF all tasks in project = 'done': UPDATE project SET status = 'shipped'
DELIMITER $$
CREATE TRIGGER trg_unlock_task_after_submission
AFTER INSERT ON submission
FOR EACH ROW
BEGIN
    IF NEW.ai_score >= 70 AND NEW.lesson_id IS NOT NULL THEN
        UPDATE kanban_task kt
        JOIN project p ON kt.project_id = p.project_id
        SET kt.status = 'in_progress'
        WHERE kt.lesson_id = NEW.lesson_id  -- matches the lesson just submitted
          AND kt.status    = 'locked'       -- only unlock currently locked tasks
          AND p.user_id    = NEW.user_id;   -- only the submitter's own projects
    END IF;
END$$
DELIMITER ;
NEW keyword
Inside a trigger, NEW refers to the row being inserted or updated. NEW.ai_score is the score of the just-inserted submission. OLD (for UPDATE/DELETE triggers) is the row before the change.

Trigger 2 — Enforce team size limit

DELIMITER $$
CREATE TRIGGER trg_enforce_team_size
BEFORE INSERT ON team_member
FOR EACH ROW
BEGIN
    DECLARE current_members INT;
    DECLARE team_limit      INT;

    SELECT COUNT(*) INTO current_members
    FROM team_member WHERE team_id = NEW.team_id;

    SELECT max_members INTO team_limit
    FROM team WHERE team_id = NEW.team_id;

    IF current_members >= team_limit THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Team has reached its maximum member capacity.';
    END IF;
END$$
DELIMITER ;
SIGNAL SQLSTATE
SIGNAL raises a custom error from inside a trigger. The code '45000' is the user-defined exception class. MySQL cancels the INSERT and propagates the error up to the application — caught by the global error handler in server.js.

Trigger 3 — Auto-ship project when all tasks complete

DELIMITER $$
CREATE TRIGGER trg_auto_ship_project
AFTER UPDATE ON kanban_task
FOR EACH ROW
BEGIN
    DECLARE total_tasks INT;
    DECLARE done_tasks  INT;

    SELECT COUNT(*),
           SUM(CASE WHEN status = 'done' THEN 1 ELSE 0 END)
    INTO total_tasks, done_tasks
    FROM kanban_task WHERE project_id = NEW.project_id;

    IF total_tasks > 0 AND total_tasks = done_tasks THEN
        UPDATE project SET status = 'shipped'
        WHERE project_id = NEW.project_id AND status != 'shipped';
    END IF;
END$$
DELIMITER ;
The status != 'shipped' guard
Without this guard, marking a task done on an already-shipped project would fire the UPDATE again unnecessarily. The guard prevents a redundant write and avoids triggering any further downstream cascade effects.

Transactions — ACID Guarantees

A transaction is a group of SQL statements treated as one indivisible unit. Either all succeed (COMMIT) or none do (ROLLBACK).

ACIDPropertyWhat it means
AAtomicityAll-or-nothing. No partial writes survive.
CConsistencyThe database moves from one valid state to another. Constraints are never violated mid-transaction.
IIsolationConcurrent transactions don't see each other's uncommitted work.
DDurabilityOnce committed, data survives crashes and power failures.

Transaction 1 — Enroll a new user atomically

START TRANSACTION;

    -- Step 1: Create the user
    INSERT INTO users (f_name, l_name, email, password_hash, role, sub_tier)
    VALUES ('Christopher', 'Clark', 'christopher.clark@devpath.io',
            '$2b$10$...', 'learner', 'pro');

    -- Step 2: Enroll in Full-Stack Web Dev
    -- LAST_INSERT_ID() returns the user_id just generated by Step 1
    INSERT INTO enrollment (user_id, course_id, progress)
    VALUES (LAST_INSERT_ID(), 1, 0);

COMMIT;
-- Either both rows exist or neither does.

Transaction 2 — Create team and assign to project

START TRANSACTION;
    INSERT INTO team (team_name, max_members) VALUES ('Team Apex', 4);
    INSERT INTO team_member (user_id, team_id) VALUES (5, LAST_INSERT_ID());
    INSERT INTO team_member (user_id, team_id) VALUES (3, LAST_INSERT_ID());
    UPDATE project
    SET    team_id = (SELECT team_id FROM team WHERE team_name = 'Team Apex')
    WHERE  project_id = 5;
COMMIT;

Transaction 3 — Rollback on invalid data

START TRANSACTION;

    -- ai_score=110 violates CHECK(ai_score BETWEEN 0 AND 100)
    INSERT INTO submission (user_id, sub_date, lesson_id, ai_score, feedback)
    VALUES (3, '2025-03-25 12:00:00', 5, 110, 'Score exceeds maximum.');
    -- MySQL raises a CHECK constraint error here

ROLLBACK;
-- The table is exactly as it was before START TRANSACTION. Zero partial writes.
ROLLBACK = time machine
ROLLBACK undoes everything since START TRANSACTION. In banking: if debit succeeds but credit fails, ROLLBACK removes the debit. No half-completed operations survive — this is what makes transactional databases trustworthy.

Startup and middleware

Why a pool?
Opening and closing a MySQL connection for every HTTP request is slow (TCP handshake + auth each time). A connection pool keeps connections permanently open and hands them out on demand. When a route finishes, the connection is returned to the pool for reuse.

authenticate middleware — full walkthrough

const authenticate = async (req, res, next) => {
    // Authorization header looks like: "Bearer eyJhbGci..."
    const token = req.headers.authorization?.split(' ')[1];
    if (!token) return res.status(401).json({ error: 'Unauthorized' });
    try {
        // jwt.verify() checks the signature AND expiry simultaneously
        const decoded = jwt.verify(token, JWT_SECRET);

        // Extra safety: confirm user still exists (catches admin deletions)
        const [rows] = await pool.query(
            'SELECT user_id FROM users WHERE user_id = ?', [decoded.id]
        );
        if (rows.length === 0)
            return res.status(401).json({ error: 'Account no longer exists.' });

        req.user = decoded;  // attach { id, role } so route handlers can use it
        next();              // pass control to the actual route handler
    } catch {
        res.status(401).json({ error: 'Invalid or expired token' });
    }
};

Login route — complete flow

const loginSchema = z.object({
    email:    z.string().email(),   // Zod rejects "notanemail" before DB is touched
    password: z.string().min(1)
});

app.post('/api/auth/login', async (req, res, next) => {
    try {
        const { email, password } = loginSchema.parse(req.body); // throws if invalid

        const [users] = await pool.query(
            'SELECT * FROM users WHERE email = ?', [email]
        );                              // ? = parameterised — prevents SQL injection
        if (users.length === 0)
            return res.status(401).json({ error: 'Invalid credentials' });

        const match = await bcrypt.compare(password, users[0].password_hash);
        if (!match)
            return res.status(401).json({ error: 'Invalid credentials' });

        const token = jwt.sign(
            { id: users[0].user_id, role: users[0].role },
            JWT_SECRET, { expiresIn: '24h' }
        );
        res.json({ token, user: { /* profile fields */ } });
    } catch (err) { next(err); }  // pass to global error handler
});

Global error handler

app.use((err, req, res, next) => {
    if (err instanceof z.ZodError)
        return res.status(400).json({ error: 'Validation failed', details: err.errors });
    res.status(500).json({ error: err.message || 'Internal Server Error' });
});

Every route ends its catch block with next(err). All errors — Zod validation failures, MySQL errors, token errors — flow here. This single handler normalises them into consistent { "error": "..." } JSON. The frontend can always handle errors the same way regardless of origin.

What is a Single Page Application?

A SPA loads one HTML page once, then dynamically swaps content without full page reloads. Navigation changes the URL hash (the #platform part) without triggering a new HTTP request. JavaScript listens for hash changes and shows/hides content panels accordingly.

Browser loads index.html ONCE │ ├── User clicks [Showcase] tab │ → window.location.hash = '#showcase' │ → 'hashchange' event fires │ → JS: hide #platform-panel │ → JS: show #showcase-panel │ → fetch('/api/showcase/queries') → render list │ (no page reload — instant) │ └── User clicks [Platform] tab (logged in) → hash = '#platform' → fetch('/api/profile/me', { headers: { Authorization: 'Bearer ...' }}) → render dashboard from returned JSON

localStorage — persisting state between page loads

// After successful login:
localStorage.setItem('brio_token', token);

// Before any authenticated API call:
const token = localStorage.getItem('brio_token');
fetch('/api/profile/me', {
    headers: { Authorization: `Bearer ${token}` }
});

// On logout:
localStorage.removeItem('brio_token');

The critical insight

Security Layers — Defence in Depth

Brio has seven independent security layers. If one fails, the others hold.

Quick Reference Cheat Sheet

SQL Query Anatomy

SELECT   col, COUNT(*), AVG(col2)    -- what to return
FROM     table_a a
JOIN     table_b b ON a.id = b.fk   -- combine tables
WHERE    col3 = 'value'              -- filter BEFORE grouping
GROUP BY col                          -- collapse rows into groups
HAVING   COUNT(*) > 3                -- filter AFTER grouping
ORDER BY col DESC                    -- sort
LIMIT    10;                          -- max rows

JOIN Types

TypeReturns
INNER JOIN / JOINOnly rows with a match on both sides
LEFT JOINAll left rows + matched right rows (NULL if no match)
RIGHT JOINAll right rows + matched left rows (NULL if no match)

WHERE vs HAVING

ClauseWhen it filtersCan use aggregate functions?
WHEREBefore GROUP BY — on individual rowsNo
HAVINGAfter GROUP BY — on groupsYes (COUNT, AVG, SUM…)
